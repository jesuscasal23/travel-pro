1. User Groups & Permissions

👤 Guest (Unauthenticated)

Can plan & generate trips without signing up. Cannot save trips to dashboard or access profile.

Can Access

  • Landing page /
  • Onboarding /onboarding
  • Plan questionnaire /plan
  • Trip view /trip/[id] (incl. guest mode)
  • Trip edit /trip/[id]/edit
  • Trip summary /trip/[id]/summary
  • Shared view /share/[token]
  • Auth pages (signup/login/forgot/reset)

Cannot Access

  • Dashboard /dashboard → redirect to login
  • Profile /profile → redirect to login
🔑 Authenticated User

Full access. Trips linked to profile. Can save, edit, share, optimize, and manage account.

Additional Access

  • Dashboard /dashboard
  • Profile settings /profile
  • Trip ownership operations (PATCH/DELETE)
  • Flight optimization
  • Share link generation
  • GDPR data export + account delete

Plan Flow

Skips onboarding steps 1-2 (profile already saved). Starts at step 3 (destination).

🌐 Share Link Viewer

Read-only access to a shared itinerary via token. No auth required. Sees growth CTA.

Can See

  • Full itinerary (route, days, budget, visa, weather)
  • "Start free" CTA → /signup

Cannot Do

  • Edit itinerary
  • Generate activities
  • Access any other page (unless navigating away)

7. API Route Details — Data Flow

Generation Pipeline (the core product flow)

Client (/plan)                      Server                               External
─────────────                      ──────                               ────────

1. POST /api/generate/select-route ──▶ selectRoute()
   { profile, tripIntent }             │ single-city? → return destination immediately
                                       │ multi-city?  → Haiku prompt ──────▶ Anthropic Haiku
                                       │                                     (route-selector.ts)
                                       ◀── { cities: CityWithDays[] }

2. POST /api/v1/trips              ──▶ prisma.trip.create() ───────────▶ PostgreSQL
   { tripType, region, dates... }      │                                  (Trip record)
                                       ◀── { trip: { id, ... } }

3. POST /api/v1/trips/:id/generate ──▶ createGeneratingRecord() ──────▶ PostgreSQL
   { profile, cities }                 │                                  (Itinerary, status=generating)
                                       │ generateRouteOnly() ─────────▶ Anthropic Haiku
                                       │   (route + budget, no activities)  (route-only.ts prompt)
                                       │ activateGeneratedItinerary() ─▶ PostgreSQL
                                       │                                  (Itinerary, status=complete)
   ◀── SSE stream:                     │
     { stage: "route", pct: 20 }       │
     { stage: "done",  pct: 100 }      │

4. POST /api/v1/trips/:id/         ──▶ generateCityActivities() ──────▶ Anthropic Haiku
       generate-activities              │                                  (city-activities.ts prompt)
   { profile, cityId }                 │ prisma.itinerary.update() ────▶ PostgreSQL
                                       │   merge activities into days      (Itinerary.data updated)
                                       ◀── { days: TripDay[] }

5. POST /api/v1/enrich/visa        ──▶ enrichVisa() ──────────────────▶ Static dataset
   { nationality, route }              │                                  (visa-index.ts, 199×227)
                                       ◀── { visaData: VisaInfo[] }

6. POST /api/v1/enrich/weather     ──▶ enrichWeather() ───────────────▶ Open-Meteo Archive API
   { route, dateStart }                │                              ──▶ Upstash Redis (cache 7d)
                                       ◀── { weatherData: CityWeather[] }

Flight Optimization Pipeline

Client (/trip/:id)                  Server                               External
─────────────                      ──────                               ────────

POST /api/v1/trips/:id/optimize ──▶ optimizeFlights()
{ homeAirport, route, dates,        │
  travelers }                        │ 1. Generate all feasible day assignments
                                     │    (permutations of minDays..maxDays per city)
                                     │
                                     │ 2. Collect unique flight legs needed
                                     │
                                     │ 3. Parallel search all legs ──────▶ Amadeus Flight API
                                     │    (with Redis cache, 2h TTL)   ──▶ Upstash Redis
                                     │
                                     │ 4. Find cheapest feasible combo
                                     │
                                     ◀── { skeleton: { legs, totalFlightCost, ... } }

Owner clicks "Share"                Server
──────────────────                 ──────

Cascade Rules:
  Profile DELETE → Trip DELETE → Itinerary DELETE → ItineraryEdit DELETE
  Trip DELETE → AffiliateClick SET NULL (tripId)

Itinerary.data JSON structure:
  {
    route:       CityStop[]
    days:        TripDay[]         (may be empty if route-only generation)
    budget:      TripBudget
    visaData?:   VisaInfo[]        (enriched client-side)
    weatherData?: CityWeather[]    (enriched client-side)
    flightLegs?: ItineraryFlightLeg[]  (from Amadeus optimization)
    flightBaselineCost?: number
  }

Itinerary Versioning

Every edit that modifies the itinerary data creates a new Itinerary record at version + 1. The previous version is deactivated (isActive = false) in an atomic transaction. This means tripId is not unique on Itinerary — never use upsert { where: { tripId } }.

Generation Status Lifecycle

  pending ──▶ generating ──▶ complete
                  │
                  └──▶ failed  (timeout, content filter, or stale cleanup cron)

13. Simplification Opportunities

1. Phase 0 generation route (POST /api/generate)
This legacy route is from the Phase 0 demo. The Phase 1 flow uses /api/v1/trips/:id/generate instead. The route selection endpoint /api/generate/select-route is still actively used by the plan page, but the main /api/generate route appears to be dead code.
Action: Verify no clients call /api/generate and remove it.
2. Duplicate state: Zustand store vs Database
The Zustand store persists the entire itinerary object to localStorage, which duplicates what's stored in the database Itinerary.data. For authenticated users, the DB is the source of truth, but the store is also kept in sync. This creates potential consistency issues.
Action: Consider making Zustand the cache-only layer for the current session and always fetching from DB for authenticated users.
3. Guest trip → auth linking gap
When a guest creates a trip and later signs up, the trip's profileId remains null. There is no mechanism to retroactively link guest trips to a new account. This means guest trips are essentially orphaned.
Action: Either implement trip claiming (via localStorage tripId after signup) or explicitly communicate that guest trips are temporary.
4. Two-step generation (route-only → per-city activities)
The new generation flow splits into generateRouteOnly() + generateCityActivities() (per city). This requires N+1 Anthropic API calls (1 for route + N for cities). The old pipeline did it in one call with higher token budget.
Trade-off: Better UX (progressive loading) vs. higher API cost and complexity. Consider whether the progressive loading is worth the extra calls.
5. Enrichment as separate API routes
Visa and weather enrichment are separate POST endpoints (/api/v1/enrich/visa, /api/v1/enrich/weather) called from the client. Since visa is purely static data (no external API), it could be done client-side. Weather could be folded into the generation pipeline.
Action: Move visa lookup to client-side utility. Consider whether weather enrichment needs its own route or could be part of generation.
6. Unused DB models
The Prisma schema mentions 5 models currently, but the CLAUDE.md references 8 models (including Experiment, ExperimentAssignment, AnalyticsEvent). Check if those 3 were removed from the schema or if they are unused but still defined.
Action: Audit schema for unused models and remove them.
7. Onboarding duplication
The onboarding page (/onboarding) and plan page steps 1-2 (guest mode) collect the same data (nationality, airport, travel style, interests). Both write to the same Zustand fields. The plan page effectively duplicates onboarding for guests.
Action: Consider whether /onboarding is still needed or if the plan page's guest flow makes it redundant.
8. Affiliate click tracking
AffiliateClick records are created but there's no UI to view or analyze them. The affiliate links themselves are static URL patterns, not actual partner integrations with revenue tracking.
Action: Either build analytics dashboard for affiliate data or simplify by removing the tracking if it's not being used.
9. Resend email integration
Email templates exist for welcome and itinerary-ready notifications, but it's unclear where sendWelcomeEmail() and sendItineraryReadyEmail() are actually called from in the current flow. They may not be wired up.
Action: Verify email sending is actually triggered. If not, either wire it up or remove the dead code.
10. Test coverage gaps (critical paths)
The core revenue path (trip summary → affiliate links → booking) has zero test coverage. Profile/GDPR operations (data export, account delete) are also untested. The AI pipeline (prompt assembly, JSON parsing, content filter retry) has no unit tests.
Action: Prioritize tests for: (a) AI pipeline parsing/validation, (b) profile GDPR operations, (c) summary/share flow.
