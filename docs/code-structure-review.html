Architecture review · repository scan on 2026-04-04

The app is not chaotic, but its seams are getting expensive.

Fichi already has a recognizable architecture: pages call client hooks, hooks call route handlers, route handlers call feature services, and feature services call Prisma or external APIs. The problem is that the same product concepts are implemented repeatedly across those layers, so every feature change requires synchronized edits in several places.

What is already good

  • Feature folders are real and not cosmetic.
  • The route layer has a standard wrapper via apiHandler.
  • The enrichment feature already uses factories instead of copy-paste.
  • Backend conventions are documented and mostly coherent.

What feels expensive

  • A single product concept often exists as route code, hook code, service code, query keys, and store glue.
  • Many modules are thin enough that reading the interface is almost the same work as reading the implementation.
  • Some conventions are documented, but a few important routes still bypass them.

2. Friction map

These are the areas where structural complexity is materially affecting maintainability, not just stylistic preferences.

Friction A: selection handling is duplicated across three layers

The flights and hotels selection flows are implemented as parallel universes. The route files are almost identical. The service layer repeats the same CRUD shape. The client hooks repeat the same query and mutation patterns.

Route layer .../selections/flights/route.ts and .../selections/hotels/route.ts
Service layer get/upsert/remove/mark exists twice in selection-service.ts
Client layer query hooks and mutation helpers mirror the same split again
flights ↔ hotels ↔ same control flow, renamed fields

Why this matters

Every bug fix or behavior change has to be remembered in multiple places. That multiplies risk and slows review because the important part is buried under mechanical repetition.

What this signals

The real abstraction is not "flight route" or "hotel route". The real abstraction is "selection resource with kind-specific schema and persistence rules".

What already points the way

The enrichment endpoints already use a shared factory. The selections feature should be pulled in the same direction.

Friction B: planner state is split across too many owners

Traveler preferences, trip draft fields, guest fallback behavior, server profile hydration, and trip identity are spread across stores, hooks, providers, and pages.

Current ownership map
useTripStore transient nationality, airport, interests, pace, vibes, build flags
usePlanFormStore route draft, dates, travelers, priorities, step state
useTravelerPreferences merges server profile with transient fallback
PlanPage and ProfilePage do manual hydration and manual sync back into stores
useTripIdentity reassembles trip-level inputs from trip query plus plan store plus preference hook
What the app actually needs
One planning session boundary one place owns the shape of in-progress planning state
Explicit hydration rules server profile, guest draft, and sign-out reset live in one module
Selectors and commands pages ask for "traveler preferences" or "trip draft", not raw stores
Cleaner UI pages pages become orchestration and rendering, not reconciliation engines

Friction C: conventions drift in a few important places

Area Current state Why it matters
Admin routes Some admin endpoints query Prisma directly in route files. The route layer becomes heavier and can drift from the stated "thin route" convention.
Env access Most config has a boundary, but some routes still read process.env directly. Validation and fallback behavior become inconsistent.
Naming The repo still contains both Fichi and Travel Pro terminology. That is less about architecture and more about maintenance confusion, docs drift, and product polish.
Dev routes Both _dev and %5Fdev route entrypoints exist. It is a small example of route-surface duplication that future maintainers need to mentally explain.

3. Proposal 1: selection resource deep module

This is the highest-leverage next refactor. It is localized, measurable, and it removes copy-paste from the exact area where the app currently repeats itself most aggressively.

Recommended first Low product risk High duplication payoff
Current selections model
  • Two route files with near-identical auth, validation, and response flow.
  • Two CRUD clusters in the same service file.
  • Parallel query hooks and parallel mutation logic.
  • Any third selection kind would add another full vertical slice.
Proposed selections model
  • One shared selection resource factory for route handlers.
  • One kind descriptor per selection type.
  • One client query factory and one mutation factory.
  • Kind-specific differences become data and adapters, not control-flow duplication.
Descriptor
Define the resource once per kind route path, query key, Zod schema, Prisma select, upsert strategy, booked mutation, remove mutation, serializer
↓
Server factory generate GET / PUT / DELETE / PATCH handlers from the descriptor
Client factory generate query and mutation hooks from the same descriptor
↓
Result
One deep module instead of several shallow twins callers use a small interface while the module hides auth, schemas, cache updates, Prisma specifics, and optimistic update rules

Concrete shape

  • Create selection-resource.ts with a generic descriptor type.
  • Move flight-specific and hotel-specific persistence details into separate descriptor modules.
  • Generate route handlers instead of hand-writing two route files.
  • Generate query hooks instead of keeping mirrored files for flights and hotels.

What stays kind-specific

  • Unique keys for upsert.
  • Field-level Zod schema differences.
  • Prisma model name and select shape.
  • How optimistic cart updates map a stored record back into flight or hotel arrays.

Migration sequence

Step Change Why this order
1 Extract shared route factory and descriptor types. Reduces server duplication without changing endpoint contracts.
2 Refactor the service layer into kind descriptors plus common operations. Keeps business rules behind one boundary before touching client hooks.
3 Collapse query hooks and mutation helpers onto shared factories. Lets the client side consume the new deep module cleanly.
4 Backfill boundary tests around the generated resource behavior. Replaces repeated per-kind test setup with more valuable behavior tests.

4. Proposal 2: planning session boundary

This is the more strategic refactor. It addresses the hardest-to-explain part of the app: where planning state comes from, who owns it, and how guest and authenticated flows merge.

Do second Broader blast radius High architectural payoff
Today

The planner is spread across two stores, a preference-merging hook, page-level hydration logic, and provider-level reconstruction. The model works, but it is hard to answer simple questions like "what is the current traveler profile?" or "which date values are authoritative right now?"

After refactor

One planning session module owns the lifecycle of in-progress planning. Pages consume stable selectors like traveler preferences, trip draft, and session commands. Hydration and fallback logic stop leaking into UI files.

What this boundary should own

  • Traveler preferences for both guest and authenticated users.
  • Trip draft fields such as route, dates, travelers, and priorities.
  • Hydration from profile query into local draft state.
  • Reset semantics on sign-out.
  • Derivations used by trip generation and trip identity.

What UI files should stop doing

  • Calling useTripStore.setState(...) directly.
  • Re-deriving effective nationality and airport from multiple sources.
  • Mixing planner state reconciliation with rendering.
  • Reaching into both stores plus server query results for the same user intent.

Practical target design

Create a planning-session module It can still be backed by Zustand internally, but external consumers should not need to know about multiple stores.
Expose selectors, not storage details For example: traveler preferences, trip draft, effective dates, can-generate, session reset, hydrate-from-profile.
Move merging logic out of pages PlanPage, ProfilePage, and useTripIdentity should stop being mini state managers.
Treat guest and auth as session modes That turns conditional fallback behavior into a modeled state concern instead of a repeated UI concern.

Migration sequence

Step Change Outcome
1 Introduce planning-session selectors that read from existing stores. No behavior change yet; just adds a stable facade.
2 Move profile hydration and guest fallback into the facade. Pages stop doing manual merge logic.
3 Update PlanPage, ProfilePage, and useTripIdentity to consume the facade only. The boundary becomes real and externally visible.
4 Collapse internal storage if desired. The app can then choose one store or keep multiple stores hidden behind one interface.

5. Recommendation and sequencing

The two proposals are complementary. One reduces duplication in a bounded feature. The other clarifies the ownership model of the planner.

What I would do next Start with proposal 1. It has the best ratio of impact to risk, it removes obvious repeated code immediately, and it gives the codebase a reusable pattern for future feature surfaces. Then do proposal 2 as the larger architectural cleanup once the team has confidence in the more factory-driven approach.

Why proposal 1 first

  • Localized scope.
  • Clear before/after reduction in file count and duplication.
  • Low user-facing behavior risk.

Why proposal 2 second

  • Touches more screens and more state assumptions.
  • Needs stronger behavioral confidence first.
  • Benefits from having one successful deep-module refactor already in place.

What not to do

  • Do not flatten everything into generic utilities with no domain vocabulary.
  • Do not delete the feature split.
  • Do not refactor planner state and selections at the same time.

This document is intentionally opinionated. The goal is not to describe every file. The goal is to show where the structure is costing you real maintenance effort, and how to improve it without destabilizing the product.
