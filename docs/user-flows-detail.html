4. API Call Sequence by Trip Type

Single-City API Sequence

  1. POST /api/generate/select-route
    Short-circuit: returns 1 city, no Haiku call
  2. POST /api/v1/trips DB
    Create Trip record
  3. POST /api/v1/trips/:id/generate Haiku DB
    SSE stream. Route + budget (4k tokens). Single-city prompt.
  4. POST /api/v1/enrich/visa
    Static lookup (no external call)
  5. POST /api/v1/enrich/weather Open-Meteo
    1 city weather
  6. POST /trips/:id/generate-activities Haiku
    1 city × 1 call
Total Haiku calls: 2 (1 generation + 1 activity)
Total external API calls: 3 (Haiku ×2 + Open-Meteo ×1)

Single-Country API Sequence

  1. POST /api/generate/select-route Haiku
    Select 3-6 cities within country
  2. User reviews/edits cities in Route Review
  3. POST /api/v1/trips DB
    Create Trip record
  4. POST /api/v1/trips/:id/generate Haiku DB
    SSE stream. Route + budget (10k tokens).
  5. POST /api/v1/enrich/visa
    Static lookup (1 country, deduped)
  6. POST /api/v1/enrich/weather Open-Meteo
    N cities weather
  7. POST /trips/:id/generate-activities Haiku
    N cities × 1 call each (on demand)
Total Haiku calls: 2 + N (route + gen + N activities)
Total external API calls: 2 + 2N
For 4 cities: 6 Haiku + 4 Open-Meteo = 10 calls

Multi-City API Sequence

  1. POST /api/generate/select-route Haiku
    Select 4-7 cities across region
  2. User reviews/edits cities in Route Review
  3. POST /api/v1/trips DB
    Create Trip record
  4. POST /api/v1/trips/:id/generate Haiku DB
    SSE stream. Route + budget (10k tokens).
  5. POST /api/v1/enrich/visa
    Static lookup (multiple countries)
  6. POST /api/v1/enrich/weather Open-Meteo
    N cities weather
  7. POST /trips/:id/generate-activities Haiku
    N cities × 1 call each (on demand)
  8. POST /trips/:id/optimize Amadeus
    Flight optimization (auth only, optional)
Total Haiku calls: 2 + N (route + gen + N activities)
Total external API calls: 2 + 2N + Amadeus
For 5 cities: 7 Haiku + 5 Open-Meteo + Amadeus = 13+ calls

5. Observations, Gaps & Simplification Opportunities

Questions asked per trip type — do we ask too much or too little?

Single-City: Route Review is skipped, but should we ask more?
For single-city trips, the user picks a city and goes straight to generation. There is no chance to specify neighborhoods, activity preferences for that specific city, or accommodation area. For a 7-day single-city trip in Tokyo, the user has no input between "I want to go to Tokyo" and the full generated plan. Consider adding a lightweight customization step (preferred neighborhoods, must-see attractions, dietary preferences).
Single-Country: No country-level context gathered
When selecting a single country, we only ask for the country name. We don't ask:
  • Are there specific cities the user wants to include/exclude?
  • Does the user prefer coastal vs. inland?
  • Is there a specific region within the country they prefer?
The Route Review step partially addresses this (user can edit cities), but by then the AI has already made choices. Pre-generation input could improve results.
Multi-City: Region is very coarse
"Southeast Asia" or "Western Europe" covers dozens of countries. The user has no way to express preferences like "I want beach + culture" or "avoid countries X and Y" before the AI picks cities. The Route Review allows post-hoc editing, but better pre-selection input would reduce AI wasted tokens.
What's working well: The Route Review step for multi-city/single-country is genuinely useful — users can customize the AI's suggestions. The drag-drop reordering and day adjustment are well-tested.

Missing views / functionality

No city preview/details before commitment
In the Route Review step, cities are shown as names with day counts. There's no thumbnail image, brief description, or reason why the AI picked that city. Users are making decisions about cities they may not know. A mini-card with 1-line description + image would help.
No "why this city?" explanation from the AI
The route selector prompt asks Haiku to pick cities, but the response only contains city metadata (name, country, lat/lng, minDays, maxDays). There's no reason field explaining why a city was selected. Adding this to the CityWithDays schema would help users evaluate the route.
Single-city edit page is nearly useless
The edit page (/trip/[id]/edit) for single-city trips only has 1 city. The only meaningful action is adjusting day count (which doesn't change anything since all days are in that city). The "add city" button effectively converts the trip into a multi-city trip, which changes the tripType semantics silently. Consider either hiding the edit button for single-city or redesigning it to focus on activity customization.
Flight optimization only for authenticated users + multi-leg
Single-city trips have a single round-trip flight, but there's no flight price lookup for them. Amadeus optimization requires owner auth and is only useful for multi-leg routes. For single-city, a simple "search flights" link to Skyscanner would be more appropriate than the optimization endpoint.
flexibleDates field is collected but impact is unclear
The flexibleDates boolean (±3 days) is stored and sent to the API, but it's unclear how the generation prompts use it. The Amadeus flight search doesn't use flexible dates either. Verify this field actually affects output, or remove it.
Guest trip cannot be linked to account after signup
A guest creates a trip with profileId: null. If they later sign up, the trip remains orphaned — it won't appear in their dashboard. The currentTripId is in Zustand, so a post-signup migration could be implemented, but it currently isn't.

Single-Country vs Multi-City — are they different enough?

These two flows are nearly identical.
The only differences between single-country and multi-city are:
  • Destination picker: CountryCombobox vs region SelectionCard
  • Prompt constraint: "cities within {country}" vs "cities across {region}"
  • City count: 3-6 vs 4-7
Everything else (route review, generation, edit, summary, APIs) is identical. Consider whether these warrant separate trip types or could be unified with a "scope" parameter (country vs region).
